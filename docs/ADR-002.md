# ADR-002: Tenant Isolation, Release Concurrency, and DataSource Safety (Preview-only Mocks)

## Status
Proposed → Accepted → **Implemented**

## Date
2026-03-01

## Context
We have an AI-powered low-code platform where tenant apps are versioned JSON metadata (FormDoc) rendered by a compiled React shell. We already support:

- Immutable version publishing + mutable channel pointers (preview/prod)
- Client-side and server-side schema + governance validation
- Migrations (schemaVersion) and a secure runtime action system
- Shell remote mode fetch with ETag caching

As we move toward SaaS for external customers, the current publish spine lacks three critical properties:

1) **Tenant isolation:** `appKey` is currently effectively global. Without tenant partitioning, we risk cross-tenant collisions and data leaks.

2) **Promotion safety:** Release pointers are mutable. Concurrent promotions can overwrite each other unless we enforce optimistic concurrency (If-Match with ETag).

3) **DataSource safety:** If metadata can define REST datasources with arbitrary URLs, runtime may call untrusted endpoints directly, leaking data and violating governance. We also need strict policy around `mock` datasources.

We want a simple policy decision:
- **Mock datasources are allowed in `preview` only, never in `prod`.**

## Decision
### A) Tenant isolation becomes first-class
All metadata API operations and storage are partitioned by `tenantId` as a first-class key.

**API routes (new canonical):**
- `POST /v1/tenants/{tenantId}/apps/{appKey}/versions`
- `GET  /v1/tenants/{tenantId}/apps/{appKey}/versions/{versionId}`
- `GET  /v1/tenants/{tenantId}/apps/{appKey}/versions?limit=50`
- `PUT  /v1/tenants/{tenantId}/apps/{appKey}/releases/{channel}`
- `GET  /v1/tenants/{tenantId}/apps/{appKey}/releases/{channel}`

**Backward compatibility:**
- Existing non-tenant routes continue to work temporarily by mapping to `tenantId="default"` and adding a deprecation warning header (e.g., `X-Deprecated-Route: true`).

**Mongo documents must include `tenantId`:**
- `app_versions`: `{ tenantId, appKey, versionId, ... }`
- `app_releases`: `{ tenantId, appKey, channel, ... }`
- `audit_events`: `{ tenantId, appKey, ... }`

**Indexes:**
- `app_versions`: unique `(tenantId, appKey, versionId)` and unique `(tenantId, appKey, contentHash)` (optional dedupe)
- `app_releases`: unique `(tenantId, appKey, channel)`
- `audit_events`: index `(tenantId, appKey, at desc)`

### B) Release promotions require optimistic concurrency
Release pointer updates (`PUT /releases/{channel}`) must enforce optimistic concurrency using ETags.

- `GET /releases/{channel}` returns an ETag derived from current pointer (e.g., weak ETag `W/"{versionId}"`).
- `PUT /releases/{channel}` requires `If-Match` to equal the current ETag (except first creation).
- On mismatch: return `409 Conflict` with the current pointer in response body.

Client behavior:
- `publish-client.promoteRelease()` reads the latest ETag and supplies `If-Match`.
- If `409`, the client surfaces “Release moved; refresh version history.”

### C) DataSource safety: preview-only mocks; prod forbids unsafe REST
Governance becomes channel-aware for promotion and publish workflows:

**Policy:**
- Channel `prod`:
  - `mock` datasources are **rejected**
  - `rest` datasources are rejected unless:
    - URL is **relative** (same-origin) OR
    - host is in an explicit **allowlist**
- Channel `preview`:
  - `mock` datasources are **allowed**
  - `rest` datasources allowed (still governed by general validation constraints)

Enforcement points:
- On `POST /versions`: validate under **preview policy** (or “neutral policy”) to allow iteration.
- On `PUT /releases/prod`: re-run governance under **prod strict policy**. Promotion fails if doc violates prod policy.

Implementation detail:
- Governance validators (TS + C#) accept an optional `GovernanceContext { channel }` argument.
- This is a hard gate for prod release promotion.

### D) Correlation IDs for traceability
All API requests should include a correlation ID for audit and troubleshooting.

- API middleware ensures `X-Correlation-Id` exists (generate if missing).
- Include correlation ID in response headers.
- Persist correlation ID into `audit_events.meta`.

Client behavior:
- publish-client attaches `X-Correlation-Id` (generated per user action) and logs it in Studio/Shell consoles.

## Options Considered
### Option 1: Encode tenant into appKey (Rejected)
Example: appKey `"tenantA:expenseApp"`.
- Pros: fewer route changes.
- Cons: leaky abstraction; harder indexing; error-prone; encourages unsafe parsing and accidental cross-tenant access.

### Option 2: No optimistic concurrency (Rejected)
- Pros: faster to ship.
- Cons: unsafe in real usage; race conditions can silently override releases, causing production incidents.

### Option 3: Allow mock datasources in prod (Rejected)
- Pros: simpler demos.
- Cons: undermines governance and realistic production behavior; encourages shipping fake connectivity to real users; complicates support and compliance.

### Option 4: Proxy-only REST datasources (Deferred)
- Pros: best security model (no arbitrary client-side calls).
- Cons: needs backend connection registry and secret management. We will evolve towards this later (recommended), but we still enforce strict allowlists now.

## Consequences
### Positive
- Strong tenant isolation guarantees (partition keys + indexes).
- Safe promotion prevents accidental overwrites.
- Production channel is protected from unsafe network behavior and mock usage.
- Better operability via correlation IDs and audit logs.

### Negative / Costs
- Requires route + client updates and short-term backward compatibility handling.
- Requires governance to be context-aware (channel).
- Requires more test coverage for concurrency and policy enforcement.

## Implementation Notes
1) **Route migration**
   - Implement new tenant routes.
   - Keep old routes mapped to tenantId="default" with deprecation headers.
   - Update publish-client, then Studio/Shell.

2) **Data migration**
   - On read: if tenantId missing, treat as "default".
   - Optional offline script to backfill tenantId="default".

3) **ETag + If-Match**
   - ETag for releases should be stable and change whenever pointer changes (versionId is sufficient).
   - If-Match required for updates except when no pointer exists.

4) **Governance policy**
   - Add `channel` to governance validation context.
   - Add explicit checks:
     - prod rejects any `DataSourceDef.kind == "mock"`
     - prod rejects `rest` datasources unless relative or allowlisted
   - Add unit tests in both TS and C# validator ports.

5) **Correlation ID**
   - Middleware ensures header exists.
   - Add to audit log meta for publish/promote/fetch.

## Rollout Plan
Phase 1:
- Implement tenant routes + server fallback mapping to default tenant.
- Update publish-client, Studio, Shell to use tenant routes with default tenant.

Phase 2:
- Implement If-Match promotion checks; update client logic.
- Add tests and docs.

Phase 3:
- Enable channel-aware governance on prod promotion:
  - Block `mock` in prod
  - Block unsafe `rest` URLs in prod
- Update docs and CODEBASE_GUIDE.

Phase 4:
- Deprecate and eventually remove non-tenant routes after consumers migrate.

## Acceptance Criteria
- Two tenants can publish identical appKey without collisions.
- Concurrent promotions produce 409 conflicts rather than overwriting.
- Promoting to prod fails if mock datasources exist.
- Promoting to prod fails if rest datasource URL is not relative or allowlisted.
- Every publish/promote/fetch is traceable via X-Correlation-Id in audit logs.
